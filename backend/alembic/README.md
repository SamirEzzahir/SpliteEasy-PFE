# Alembic migrations

Schema migrations for the SplitEasy backend. Alembic reads the DB URL and model
metadata from the app itself (`alembic/env.py` → `app.core.config.settings` and
`app.models.Base.metadata`), so there are no duplicated connection strings.

Run from the `backend/` folder (local) or `/app` (container). The Makefile at the
repo root wraps the common commands against the running container.

## Commands

| Task | Local (from `backend/`) | Docker (repo root) |
|---|---|---|
| Apply all migrations | `alembic upgrade head` | `make migrate` |
| New autogenerated migration | `alembic revision --autogenerate -m "msg"` | `make migration m="msg"` |
| Roll back one | `alembic downgrade -1` | `make migrate-down` |
| Mark DB at head (no run) | `alembic stamp head` | `make stamp` |
| Show current revision | `alembic current` | — |

## Adoption model (why the baseline is empty)

Alembic was added to a database that **already had its full schema**, created at
startup by `Base.metadata.create_all` + the idempotent statements in
`app/core/migrations.py`. So `0001_baseline` is an intentional **no-op** and every
existing database is marked with `alembic stamp head` (done for the dev DB).

`create_all` + the startup migration runner are still active today, so the app
keeps working with or without Alembic. Alembic is the path forward for *new*
schema changes; the legacy runtime migrations can be retired incrementally.

## ⚠️ Known drift — review every autogenerated migration

`alembic revision --autogenerate` diffs the models against the **live** DB. The
current DB contains legacy tables that the models no longer define (e.g.
`money_sources`, `income_transfers`). Autogenerate will therefore emit
`op.drop_table(...)` for them. **Always review the generated file** and delete any
`drop_table` / `drop_column` you did not intend before running `upgrade`.

To silence a legacy table permanently, add an `include_object` filter in
`env.py`, or drop it for real once you've confirmed it's unused.
